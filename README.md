# Project Overview
This repository contains a prototype RLVR environment inspired by TerminalBench-style verifiable tasks. The goal is to show how task-aware verification, prompt-based policy rollouts, and reward shaping can be combined into a small but inspectable workflow for RL-style experimentation on LLM agents.

# Current Prototype
- Supports `exact_match` and `arithmetic` tasks with verifiable answers.
- Performs task-aware verification inside `DummyTerminalBenchEnv` before assigning rewards.
- Uses reward shaping with a success reward and per-step penalty to encourage short, correct trajectories.
- Logs rollouts to JSON so episodes can be inspected and compared across runs.
- Computes evaluation metrics overall and by task type.

# Environment Design
The environment returns observations containing the instruction, task type, and step count. Actions are text outputs generated by the model. An episode ends when the answer is verified as correct, the maximum step budget is reached, or an early stopping rule detects repeated verbose failed attempts. This keeps the prototype simple while still reflecting the structure of an RL interaction loop.

# Reward Design
The reward function gives a positive reward on success and applies a small step penalty to discourage unnecessarily long episodes. Failed intermediate steps receive a small penalty, while terminal failure receives zero reward. This setup is intentionally simple and meant to illustrate the basic reward-shaping decisions involved in an RLVR environment.

# Task Types
Two task types are currently included:
- `exact_match`, which requires the generated output to exactly match the expected answer.
- `arithmetic`, which compares normalized outputs so simple formatting differences do not cause a correct numerical answer to fail.

The verification logic is centralized in `src.env`, which makes it easier to extend the environment with additional task types later.

# Model and Framework Choice
Hugging Face TRL is a reasonable framework choice because it already provides the main building blocks needed for LLM post-training, including integration with transformer models and RL fine-tuning workflows. GRPO is a plausible algorithm choice for this setting because the tasks have verifiable outcomes, making it natural to optimize against completion-level rewards gathered from rollouts.

# Evaluation Metrics
The prototype reports:
- overall success rate
- overall average reward
- success rate by task type
- average reward by task type
- average steps per episode
- average steps per successful episode

These metrics are printed during evaluation and also saved in the rollout JSON files.

# Current Limitations
- The task set is still small and synthetic.
- The current system runs rollouts but does not yet perform full RL fine-tuning.
- Reward design is intentionally simple and has not been tuned systematically.
- Logging is JSON-based and does not yet include richer experiment tracking or visualization.

# Next Steps
