# Project Overview
This repository packages a prototype RL-from-virtual-reward (RLVR) environment inspired by TerminalBench-style verifiable tasks. It couples task-aware verification with reward shaping, rollout logging, and evaluation summaries so a take-home submission can include the code plus the rationale for the chosen components. The goal is to make each design decision—state representation, action space, rewards, metrics—explicit so reviewers can understand how the components work together without needing a long write-up beyond the README and logged JSON.

# Environment Creation
The current prototype targets the TerminalBench benchmark data (`data/eval_terminalbench_tasks.json`) and supports `exact_match` and `arithmetic` tasks through `DummyTerminalBenchEnv`. Observations expose the instruction, task type, and step count so the policy stays aware of the current challenge and progress, and the environment reconstruction of the prompt keeps the policy informed about prior attempts. Actions are the decoded next-token string sampled from the model, normalized before verification to keep formatting consistent and so the verifier has a clean input. Episodes end after successful verification, when the step budget is reached (`environment.max_episode_steps` defaults to 20), or when verbose repeated failures trigger the early-stop guard (three repeated long answers), which keeps this prototype efficient by avoiding redundant rollouts.

## Key Deliverables
1. **State/Observation Space Definition:** the agent receives the text instruction, task type label, and step count, plus any prompt context carried in `DummyTerminalBenchEnv` before verification. Prompts are rebuilt per rollout to include earlier attempts so the agent can condition on its previous outputs.
2. **Action Space Definition:** each action is the next token string generated by the policy (after decoding and trimming) that will be normalized and sent to the verifier. No separate correction or selection actions exist; the agent simply samples the next text chunk per step.
3. **Reward Function Design (minimum 2):** the prototype currently emits a positive success reward whenever verification succeeds, plus a configurable negative step penalty (from `configs/base.yaml`) each timestep. Failure reward is set to zero, so the shaped signal becomes `success_reward - step_penalty * steps`, encouraging concise, correct answers while providing both reward channels the brief asked for.
   - Each item is written to emphasize why the representation or reward matters: the observation space focuses on what the policy actually needs to decide, the action space keeps the policy grounded in text generation rather than higher-level edits, and the reward channels trade off correctness and efficiency so the environment reflects virtual reward reasoning.

The dataset also now includes a `counting` task type (e.g., “Count how many times letter 'a' appears in word 'banana'”). These tasks are still verifiable because `DummyTerminalBenchEnv` parses the instruction, counts the occurrences, and verifies that the model’s numeric output matches that count.

# Model Selection and Training Plan
- **Base model:** `Qwen/Qwen2.5-0.5B-Instruct`, chosen for its instruction-tuned capabilities and manageable compute profile. This is the checkpoint loaded in the current pipeline.
- **RL algorithm:** GRPO (Gradient-Regularized Policy Optimization) is the chosen policy optimizer because constraining updates helps stabilize learning with deterministic verification feedback.
- **Dataset:** evaluation currently runs on the five TerminalBench tasks defined in `data/eval_terminalbench_tasks.json` (two exact-match, two arithmetic, one counting), which reflects the verifiable tasks referenced in the prompt.
- **Hyperparameters (from `configs/base.yaml`):** `max_steps=100`, `batch_size=2`, `learning_rate=1e-5`, `environment.max_episode_steps=20`. TRL’s underlying optimizer (AdamW by default) handles the gradient updates unless manually overridden.
These choices follow the deliverable template: select a base model that balances capability and cost, justify GRPO for the sparse rewards, specify the dataset so reviewers know what “benchmark” means, and list the primary hyperparameters so the RL phase is reproducible.

# Evaluation Metrics
A shared helper (`src.metrics`) computes the key metrics requested in the deliverables: reward per episode (average reward), benchmark accuracy (success rate), average steps per episode, average steps per successful episode, and per-task-type success rates/average rewards. Those values provide both a global view (how good is the policy overall and how long does it take) and a task-type view (where the policy struggles). These metrics are printed at the end of every evaluation run and persisted into the rollout JSON file alongside the episode traces so someone comparing runs can pair the raw transitions with the aggregated numbers.

# Scaling Considerations
For smaller models, keep tasks simple (more `exact_match`/`counting` prompts), sharpen reward shaping (higher success reward relative to the step penalty), reduce episode lengths, and use a smaller batch/rollout group size so each gradient update stays lightweight. Conversely, larger models can handle harder tasks or longer explanations, so raise the max episode length, increase batch or rollout group sizes, and modestly lower the learning rate to keep updates stable while exploring more complex prompts. Curriculum-wise, offer simpler tasks first and add the synthetic counting/TerminalBench variations only after the policy reliably handles the easy prompts; for larger models you can interleave harder tasks sooner, while smaller models benefit from longer warm-up on exact-match questions.

# Current Limitations
- The prototype only runs on the provided TerminalBench JSON, so it covers only the tasks supplied in the dataset; adding new variations still requires hand-editing that JSON or providing a new file.
- Reward penalties, early-stopping thresholds, and rollout logging formats are static, meaning experimentation with those knobs demands changing the YAML, rerunning, and manually comparing the JSON files.
- Rollout logging emits a single JSON per run, and no downstream tooling aggregates or charts the metrics yet, so understanding long-term behavior still means manually inspecting each file.
